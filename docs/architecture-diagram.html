Production-Scale Vacation Rental Marketplace — Architecture

High-level scaling strategy for frontend, backend, storage, search, and deployment (this take-home's implementation is deliberately a thin slice of this: single listing page, one read endpoint, no auth/search/booking).

Client

Web (React/Next.js SSR)

SSR + ISR for listing pages (SEO, fast first paint). Client-side nav for search/booking flows.

Mobile apps (iOS/Android)

Native apps hitting the same public API surface.

CDN / Edge

CloudFront / Vercel Edge — static assets, cached SSR pages, image delivery, TLS termination.

↓
API Gateway

API Gateway / BFF

AuthN/Z, rate limiting, request routing to domain services, response shaping per client (web/mobile).

↓
Domain services (stateless, horizontally autoscaled)

Listings Service

CRUD for listing content, photos, amenities. Owns the listing read/write path this take-home's API is a stand-in for.

Search Service

Query parsing, ranking, geo/date/price filters. Reads from the search index, never the primary DB directly.

Booking Service

Availability, pricing, reservations. Strong consistency requirements — owns booking transactions.

Media Service

Upload handling, async image processing/optimization pipeline.

Reviews Service

Review submission, rating aggregation, moderation.

Notifications Service

Email/SMS/push, driven off the event bus, not called synchronously by other services.

↓
Data, search & async infrastructure

Primary datastore

Postgres (transactional: bookings, users, payments) with read replicas. MongoDB acceptable for listing content, which is document-shaped and read-heavy.

Cache

Redis — session storage, hot listing reads, rate-limit counters.

Search index

OpenSearch/Elasticsearch cluster, kept in sync via CDC (Debezium-style) off the primary DB — search never queries Postgres/Mongo live.

Object storage + CDN

S3 for listing photos, fronted by CDN. Uploads trigger an async resize/optimize pipeline.

Event bus / queue

Kafka/SQS — booking confirmations, search-index updates, notification triggers. Decouples services from synchronous calls.

Observability

Centralized logs, metrics, distributed tracing, alerting across every service above.

↓
Deployment

Containers on Kubernetes / ECS

Each domain service its own deployable unit, independently scaled, across multiple availability zones.

CI/CD

Automated test → build → canary/blue-green rollout per service.

Infra as code

Terraform-managed environments (staging/prod), reproducible and reviewable.
